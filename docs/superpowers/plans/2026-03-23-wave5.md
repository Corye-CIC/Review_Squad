## Chunk 1: Server & Bridge Fixes

### Task 1: Rate limiter on port 4000

**Files:**
- Modify: `services/agent-chat/server.js` (agentClients Map state + message handler)

Context: `agentClients` is a `Map<WebSocket, {agentId: string|null}>`. We need to add per-connection rate limiting — track message count in a rolling window and drop + log messages that exceed the limit. A simple token-bucket or sliding-window counter on the state object is fine. No external deps.

- [ ] **Step 1: Add rate-limit fields to connection state**

In `wssAgents.on('connection', ...)`, change:
```js
agentClients.set(ws, { agentId: null });
```
to:
```js
agentClients.set(ws, { agentId: null, msgCount: 0, windowStart: Date.now() });
```

- [ ] **Step 2: Add rate-limit check at top of message handler**

Add a `RATE_LIMIT_MAX` and `RATE_LIMIT_WINDOW_MS` constant in the Config block:
```js
const RATE_LIMIT_MAX    = 60;    // messages per window
const RATE_LIMIT_WINDOW_MS = 10_000; // 10 seconds
```

At the top of `ws.on('message', ...)`, after `if (!state) return;`, add:
```js
// Rate limiting: 60 messages per 10 s per connection
const now = Date.now();
if (now - state.windowStart >= RATE_LIMIT_WINDOW_MS) {
  state.msgCount = 0;
  state.windowStart = now;
}
state.msgCount++;
if (state.msgCount > RATE_LIMIT_MAX) {
  log(`Rate limit exceeded for ${state.agentId ?? 'unauthenticated'}`);
  return;
}
```

- [ ] **Step 3: Verify no syntax errors**

```bash

```
Expected: no output (clean parse)

- [ ] **Step 4: Commit**

```bash

git commit -m "feat: add per-connection rate limiter on port 4000 (60 msg/10 s)"
```

## Chunk 3: Shell & UX Polish

### Task 4: init-session.sh — timeout env var + idempotency guard

**Files:**
- Modify: `services/chat-bridge/init-session.sh`

Two improvements:
- `BRIDGE_STARTUP_TIMEOUT` env var (currently hardcoded 3 s / 30 iterations at 0.1 s each)
- Idempotency guard: if bridge is already alive, skip startup entirely (currently it starts fresh even if healthy)

- [ ] **Step 1: Replace hardcoded timeout with env var**

Find the poll loop config comment block and the `WAITED` loop. Replace:
```bash
# 5. Poll for ready file: 100ms intervals, 3s timeout (30 iterations)
WAITED=0
while [ "$WAITED" -lt 30 ]; do
```
with:
```bash
# 5. Poll for ready file: 100ms intervals, configurable timeout (default 3s)
BRIDGE_STARTUP_TIMEOUT="${BRIDGE_STARTUP_TIMEOUT:-3}"
MAX_ITERATIONS=$(( BRIDGE_STARTUP_TIMEOUT * 10 ))
WAITED=0
while [ "$WAITED" -lt "$MAX_ITERATIONS" ]; do
```

- [ ] **Step 2: Idempotency guard — skip room+lifecycle if room already set**

The `/room` POST is idempotent on the server side (it just sets `currentRoom`). But the `/lifecycle phase_start` fires every time init-session.sh is sourced, even mid-session. Add a guard using the ready file as a proxy:

After the `if [ "$CHAT_AVAILABLE" = true ]; then` block that does room + lifecycle, wrap the lifecycle call only:
```bash
  # Lifecycle: phase_start — only fire once per bridge process lifecycle
  # Use a session marker keyed to PID to avoid duplicate events on re-source
  SESSION_MARKER="/tmp/chat-session-${REPO_HASH}-${BRIDGE_PID}.started"
  if [ ! -f "$SESSION_MARKER" ]; then
    touch "$SESSION_MARKER"
    curl -s --max-time 1 \
      -X POST \
      -H "Content-Type: application/json" \
      -d "$(jq -n --arg event "phase_start" --arg data "$COMMAND_NAME" '{event: $event, data: $data}')" \
      "http://127.0.0.1:${CHAT_BRIDGE_PORT}/lifecycle" \
      > /dev/null 2>&1 || true
  fi
```

Note: remove the existing unconditional lifecycle curl and replace with the guarded version above. The `/room` POST can stay unconditional — it's idempotent.

### Task 5: #new-pill dismiss — return focus to message list

**Files:**
- Modify: `services/agent-chat/public/index.html` (newPill click handler)

Currently clicking `#new-pill` scrolls to bottom and hides the pill, but focus lands on `<body>`. It should return to the message list (`#messages-wrap` or `#messages`) so keyboard users can immediately scroll/read.

- [ ] **Step 1: Add tabIndex to messages-wrap and focus after dismiss**

`#messages-wrap` is a `<div>` — it needs `tabindex="-1"` to be programmatically focusable without entering tab order.

In the HTML, change:
```html
<div id="messages-wrap">
```
to:
```html
<div id="messages-wrap" tabindex="-1">
```

- [ ] **Step 2: Focus messages-wrap after pill dismiss**

In the `newPill.addEventListener('click', ...)` handler, add `wrap.focus({ preventScroll: true })` after hiding the pill:
```js
newPill.addEventListener('click', () => {
  autoScroll = true;
  newPending = 0;
  newPill.style.display = 'none';
  scrollToBottom();
  wrap.focus({ preventScroll: true });
});
```

- [ ] **Step 3: Remove default focus outline on messages-wrap (cosmetic)**

`tabindex="-1"` elements get a focus ring in some browsers. Add to CSS:
```css
#messages-wrap:focus { outline: none; }
```
